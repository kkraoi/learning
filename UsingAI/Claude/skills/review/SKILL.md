# Code Review Report

git の変更差分をコードレビューし、改善点があればプロジェクト内の
`claude-outputs/改善案-{タイトル}.md` に出力するスキル。
問題がなければファイルを生成しない。ファイルの修正は行わない。

## Before You Start

- レビュー対象を確認する（デフォルト: 現在の未コミット変更全体）
- `claude-outputs/` ディレクトリがなければ作成する

## フロー

### ステップ1: レビュー対象の確定

引数がある場合はその範囲（例: `HEAD~1`, `main..HEAD`, 特定ファイルパス）を使う。
ない場合は `git diff HEAD` と `git diff --staged` の両方を取得する。
変更がなければ「レビュー対象の変更がありません」と報告して終了する。

### ステップ2: 変更内容の収集

- git diff の出力を取得する
- 変更されたファイルの現在の状態を Read ツールで読み込む
- package.json / pyproject.toml 等から技術スタック・言語・フレームワークを把握する
- コードの分析には Code intelligence ツール（定義参照・型情報・シンボル検索など）を優先的に使用する。対応できない場合は Grep 等のテキスト検索を利用する

### ステップ3: 改善点の分析

各変更箇所について以下の優先順位で判断する：

1. **公式推奨** — 使用技術の公式ドキュメント・公式スタイルガイドが推奨する方法
2. **ベストプラクティス** — OWASP・業界標準・デファクトスタンダードなパターン
3. **プロジェクト慣例** — このプロジェクト固有の慣例（合理的なものに限る）

以下はスキップする：
- 動作上問題がなく合理的な代替案が存在しない箇所
- 純粋なスタイル好みの差（プロジェクト統一ルールがない場合）
- 改善コストが得られる効果を上回る変更

### ステップ4: 出力

改善点が**ない**場合: 「レビュー完了。改善点なし。」と一行出力して終了（ファイル出力なし）。

改善点が**ある**場合: 内容を表すタイトルを考え、以下のパスにMarkdownを出力する：

```
{プロジェクトルート}/claude-outputs/改善案-{タイトル}.md
```

各改善提案は以下の構成で記述する：

---

## [改善案のタイトル]

**対象**: `path/to/file.ts:行番号`  
**優先度**: 高 / 中 / 低

### 修正前のリスク
現状のコードで起こりうる問題・脆弱性・パフォーマンス影響など。

### 修正理由
なぜ必要か。参照した公式ドキュメント・ベストプラクティスの根拠を示す。

### 修正箇所と修正コード

**修正前:**
```lang
// 現在のコード
```

**修正後:**
```lang
// 提案するコード
```

### 修正後の効果
修正することで何が改善されるか。

---

出力後、「`claude-outputs/改善案-{タイトル}.md` に X 件の改善提案を出力しました。」と報告する。
**ファイルへの修正は行わない。**
