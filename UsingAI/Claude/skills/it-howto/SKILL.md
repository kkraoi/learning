# IT HowTo

IT技術の課題・問題に対して最適解を解説するmdファイルを生成するスキル。
エージェントはファイルを変更しない。解法・コード変更箇所を提示するだけで、実装はユーザーが行う。

## Before You Start

- 入力された課題が具体的かどうかを判断する
- 課題が曖昧な場合は、ファイルを生成する前にユーザーに確認する
- コード解析はCode Intelligenceが使用可能であれば優先する。使用できない場合はGlob・Grep・Readで代替する

## フロー

### ステップ1: 入力の意図確認

入力から以下を読み取る。不明な点があればユーザーに質問して明確にする：

- **課題の内容**: 何を解決したいか
- **対象技術・言語**: 使用言語、フレームワーク、ツール
- **現状**: 今のコードや構成（あれば）
- **目標**: どういう状態になれば解決か

以下のケースは必ずユーザーに確認してから進む：
- 課題が1文以下で技術スタックが不明
- 「なんとかして」「うまくいかない」など目標が不明瞭
- 複数の解釈が可能な表現

### ステップ2: コードベースの調査

対象ファイルやコードが存在する場合、以下の優先順で解析する（変更はしない）：

1. **Code Intelligence**: IDEのシンボル解決・型情報・参照検索が使えるなら最初に使う
2. **Glob / Grep / Read**: Code Intelligenceが対象外または利用不可の場合に使う

調査時にプロジェクト固有の慣例・パターンを記録しておく（ステップ3で使用）。

### ステップ3: タイトルの決定

mdファイル名に使うタイトルを決める。課題の内容を簡潔に表す日本語または英語のタイトル（例: `非同期エラーハンドリング`, `SQLのN+1問題`）。

### ステップ4: 解法の作成

**解答の優先順位:**

1. **公式推奨**: その言語・フレームワーク・ツールの公式ドキュメントが推奨する方法
2. **集合知のベストプラクティス**: コミュニティで広く支持されている解法・デファクトスタンダード
3. **プロジェクト慣例への準拠**: ステップ2で確認した慣例が以下の条件を満たす場合は、それに合わせた解答を提示する
   - 明確な意図が読み取れる（命名規則・構造の一貫性など）
   - セキュリティ・パフォーマンス上の問題がない
   - 公式推奨と矛盾していない、または意図的に異なる判断がある

慣例と公式推奨が矛盾する場合は両方を提示し、トレードオフをユーザーに説明する。

以下の構造でmdの内容を組み立てる：

1. **課題の概要**: 何が問題か、なぜ起きるか
2. **最適解の説明**: 採用するアプローチとその根拠（公式/慣例/ベストプラクティス のどれか）
3. **代替案**: 別の選択肢がある場合は比較を添える
4. **コード例**: 具体的なコードを提示する
   - 既存ファイルへの変更の場合は「`ファイルパス` の N行目〜M行目を以下に書き換える」と明記する
   - 新規コードの場合はそのまま記載する
5. **確認方法**: 実装後に動作確認する方法

### ステップ5: mdファイルの出力

プロンプト実行フォルダ（カレントディレクトリ）の `claude-outputs/答案-{タイトル}.md` にファイルを書き出す。

- ファイルはWriteツールで出力する
- **既存ファイル・ソースコードは一切変更しない**
- 出力後、ファイルパスをユーザーに案内する
